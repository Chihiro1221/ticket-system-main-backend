### 第一关：环境搭建（今天下午的任务）

Dapr 的核心是 Sidecar（边车模式），它重度依赖 Docker。

1.  **安装基建**：确保你的电脑上装了 **Docker Desktop**（并启动它）。
2.  **安装 Dapr CLI**：去 Dapr 官网，照着命令行安装 Dapr CLI。
3.  **初始化本地 Dapr 环境**：
    *   在终端运行：`dapr init`
    *   这会在你的 Docker 里自动拉起 3 个容器：`dapr_redis`（状态存储和消息队列）、`dapr_placement`（Actor寻址用的）、`dapr_zipkin`（链路追踪）。
    *   **成就解锁**：只要 `dapr_redis` 跑起来了，你的基础设施就搞定了！

### 第二关：核心脏活 —— Java (Spring Boot) + Actor模型（明天）

这是整个系统最值钱、面试最能吹的部分：**用单线程的 Actor 解决超卖问题，不写任何锁代码。**

1.  **建项目**：用 Spring Initializr 建一个极简的 Spring Boot Web 项目，引入 `dapr-sdk-springboot` 依赖。
2.  **定义 Actor 接口**：
    *   写一个接口 `TicketActor`，里面就一个方法：`boolean deduct(int count)`。
3.  **实现 Actor**：
    *   写一个实现类。逻辑极度简单：
        *   从 Dapr 的 State Store 里读取当前库存（`ticket_count`）。
        *   如果 $>0$，库存减 1，保存回 State Store。返回 true。
        *   如果 $\le 0$，返回 false。
    *   *精髓在于*：**你不需要写 `synchronized`，也不需要用 Redisson 分布式锁！** Dapr 运行时会自动保证对同一个商品ID的 Actor 的请求是**排队串行**处理的。
4.  **发送异步事件**：
    *   在扣减成功后，调用 Dapr 客户端的 `publishEvent` 方法，向 `ticket-pubsub` 这个主题发送一条消息（比如：`"user123 抢票成功"`）。

3.  **调用 Java Actor**：

    *   把返回结果（成功或失败）响应给前端。

# 整个架构的辨证看法

你的眼界已经跳出了“写个功能”，进入了“架构设计”的层面，这非常棒。面试官最看重的不是你用了什么技术，而是**你为什么这么设计，以及你如何解决分布式环境下的各种“不确定性”**。

我会从**辩证的角度**，结合工业级高并发系统的标准，对你的架构思路进行深度复盘和重构建议。

---

### 一、 架构整体审视：Go 网关 + Java Actor

#### 1. 关于“Go 自研网关”的辩证看法
*   **你的想法：** Go 负责 I/O，转 gRPC 调后端。
*   **我的辩证建议：** 
    *   **不要只做“转发器”：** 如果只是简单的 HTTP 转 gRPC，那它只是个 Proxy。真正的工业级网关（如 APISIX 或自研）需要承担：**全链路追踪（TraceId 生成）**、**全局限流（Rate Limiting）**、**鉴权（JWT 校验）**以及**灰度发布**。
    *   **核心亮点：** 既然用 Go，你应该利用它的 `goroutine` 实现**请求合并 (Request Collapsing)**。比如 100 个查票请求在 10ms 内到达网关，网关合并成 1 个 gRPC 请求去调后端，这才是自研网关在抢票场景下的“杀手锏”。
    *   **架构风险：** Go 与 Java 之间的 gRPC 契约（Protobuf）维护成本高。建议引入 **Dapr Service Invocation**，这样 Go 网关只需要发 HTTP/gRPC 到本地 Dapr，由 Dapr 负责服务发现和重试。

#### 2. 关于“Java Actor 扣减库存”
*   **你的想法：** Actor 实现无锁化扣减。
*   **我的辩证建议：**
    *   **粒度控制：** 你的 Actor 应该以 `TicketId`（或场次 ID）为唯一标识。每一个场次是一个独立的 Actor 实例。
    *   **内存 vs 持久化：** Actor 内部状态只在 Redis（State Store）是不够的。抢票瞬间，Actor 负载极高。你应该设计 **“预扣减”与“正式扣减”**。Actor 只管内存里的令牌（Token），真正的订单生成交给异步消息。
    *   **亮点：** 结合 Dapr 的 **Placement Service**，你可以解释如何通过一致性哈希将不同场次的抢票流量分散到不同的 Java Pod 上，实现水平扩容。

---

### 二、 深度进阶：可持续完善的完整架构设计

为了让这个项目达到“面试必杀技”的级别，你需要构建以下三个支撑体系：

#### 1. 可靠性体系（解决“数据一致性”）
*   **Outbox Pattern (发件箱模式)：** 
    *   **辩证：** 你现在是异步发消息，但如果 Actor 执行完、消息还没发出去就宕机了怎么办？
    *   **方案：** 在 Actor 状态中维护一个 `PendingEvents` 列表。只有当消息被确认（ACK）后，才从 Actor 状态中移除。这叫 **“状态驱动的消息可靠性”**。
*   **Reminder 机制：** 
    *   用于处理“订单超时”。用户抢到票没付钱，Actor 自动触发 Reminder，把票回滚。这体现了你对业务闭环的理解。

#### 2. 防护体系（解决“系统崩溃”）
*   **Sentinel / Flow Control：**
    *   在 Go 网关层接入**热点参数限流**。比如某个瞬间，10 万人抢 ID 为 1001 的票，网关直接根据 ID 限流，保护后端的 Java Actor 不被击穿。
*   **离散化处理 (Sharding)：**
    *   如果某场票量极大（如周杰伦演唱会），单个 Actor 会成为瓶颈。你可以设计“库存分片”，将 1 万张票分配到 10 个 Actor 实例中，提升并发上限。

#### 3. 可观测性与演示体系（解决“面试说服力”）
*   **实时大屏 (Dashboard)：**
    *   **技术栈：** Go 收集 QPS 数据 -> Prometheus -> Grafana。
    *   **面试演示：** 在前端做一个“压力测试”按钮，点击后后端启动 100 个线程并发抢票，Grafana 曲线瞬间拉升，显示成功率 100%，错误率 0。这比你说一万句都管用。
*   **LLM 接入点（锦上添花）：**
    *   **不要为了 AI 而 AI。** 
    *   **场景建议：** “智能排队助手”。当用户没抢到票时，将当前用户的抢票行为特征发给大模型，让它分析用户是否为“疑似黄牛”，或者根据用户喜好推荐相似的场次。

---

### 三、 最终架构设计图（逻辑视图）

1.  **流量入口层：** 前端（Vue/React）+ 压测控制台。
2.  **高性能网关层 (Go)：** 
    *   职责：限流、鉴权、请求合并、Dapr 副作用注入。
3.  **核心业务层 (Java Actor + Dapr)：**
    *   **Virtual Actor：** 处理库存扣减、状态维护。
    *   **Dapr Pub/Sub：** 异步驱动订单落地。
    *   **Dapr State Store：** Redis 缓存 Actor 状态。
4.  **持久化存储层：** 
    *   MySQL：最终账本。
    *   Redis：高性能计数器。
5.  **AI & 监控层：** 
    *   Prometheus/Grafana（QPS 监控）。
    *   LLM Agent（用户画像与防作弊分析）。

### 四、 给你的辩证建议（防坑指南）

1.  **Dapr 性能开销：** 记住，Dapr Sidecar 会带来 1-2ms 的延迟。在面试中，如果面试官问“为什么不直接用 Java 写”，你要回答：**“为了解耦基础设施（状态管理、消息传递），实现多语言混合架构（Go 网关的高性能 + Java 的复杂逻辑处理）”**。
2.  **数据库压死：** 异步写库虽然好，但如果消息积压太严重，MySQL 还是会挂。你需要实现**“背压 (Backpressure)”**，当消息积压到一定阈值，Actor 应该停止接收新请求。
3.  **自研网关的必要性：** 面试官可能会问为什么不用现成的 Kong 或 Nginx。你的理由是：**“我们需要在网关层做深度的业务逻辑预处理（如针对特定 TicketId 的请求合并），这是通用网关难以实现的。”**

**目标：** 解决抢票系统中“票去了哪里”和“没付钱怎么办”的问题。

1.  **实现 Actor Reminder (订单超时释放)：**
    *   **逻辑：** Actor 扣减库存后，立即注册一个 15 分钟的 `Reminder`。
    *   **闭环：** 如果 15 分钟内未收到“支付成功”状态更新，Actor 自动回滚 Redis 库存。
    *   **面试价值：** 证明你考虑了业务全生命周期，而不是只管抢。
2.  **实现事务性发件箱 (Outbox Pattern) 的替代方案：**
